> **목적**: Gemma 4 도입 배경, 기술 판단 근거, 시스템별 적용 방향 공유

## 1. 왜 Gemma 4로 전환하는가

### 기존 구조의 문제

가재1호기(SER5)는 OpenAI GPT-4o mini API + Brave Search API 조합으로 운영되고 있었습니다. 이 구조에서 두 가지 심각한 문제가 발견되었습니다.

**데이터 정확성 문제**: GPT-4o mini가 Brave Search로 웹 검색해서 가져온 투자 데이터가 심각하게 부정확했습니다. MSTR 주가는 실제 대비 25% 틀렸고, BTC 보유량은 99% 틀렸습니다. LLM이 웹 검색 결과를 "해석"하는 과정에서 오래된 기사를 참조하거나 숫자를 혼동하는 구조적 문제였습니다.

**비용 구조 문제**: GPT-4o mini(월 $20-25)와 Brave Search(월 $10-15)를 합치면 월 $30-40(약 ₩40,000-50,000)의 고정 API 비용이 발생합니다. 24시간 무인 서버 운영에 매달 유료 API 비용이 나가는 구조는 장기적으로 비효율적입니다.

### Gemma 4가 해결하는 것

2026년 4월 2일, 구글 딥마인드가 Gemma 4를 Apache 2.0 오픈소스 라이선스로 공개했습니다. 이 모델은 세 가지 근본적인 변화를 가져옵니다.

**첫째, GPT-4o mini보다 모든 벤치마크에서 우월합니다.**

| 벤치마크 | GPT-4o mini | Gemma 4 E4B | Gemma 4 26B MoE |
|---------|:-----------:|:-----------:|:---------------:|
| MMLU Pro (지식/추론) | 57% | 69.4% | 82.6% |
| AIME 2026 (수학) | ~15% | 42.5% | 88.3% |
| LiveCodeBench (코딩) | 32.5% | 52.0% | 77.1% |
| MMMU Pro (멀티모달) | 34% | 52.6% | 73.8% |
| 비용 | 월 $20-25 | **무료** | **무료** |

E4B(현재 16GB에서 사용 가능)만으로도 GPT-4o mini를 전면 대체하면서 성능이 더 좋습니다. RAM을 32GB로 업그레이드하면 26B MoE를 사용할 수 있는데, 이 모델은 Claude Haiku급 성능을 무료로 로컬에서 제공합니다.

**둘째, 데이터 수집 방식이 근본적으로 바뀝니다.** LLM이 웹 검색해서 데이터를 가져오는 기존 방식 대신, Python 스크립트가 신뢰할 수 있는 API를 직접 호출하고 Gemma 4는 수집된 데이터를 가공/판단/문장 생성하는 역할만 합니다.

**셋째, 유료 API 의존도가 0이 됩니다.** OpenAI, Brave Search, Gemini 등 모든 유료 API를 삭제하고, Claude Max 구독 외에 추가 비용이 없는 구조가 됩니다.

## 2. 기술적 판단 근거 — 왜 Gemma 4인가

### 다른 선택지와의 비교

**Gemini Flash API**: 토큰당 $0.075로 저렴하지만, Gemma 4가 무료(₩0)인 이상 비용 경쟁이 되지 않습니다. Claude Max는 이미 구독 중이라 추가 비용이 없고, Gemma 4도 무료이므로 중간에 유료 API를 끼울 자리가 없습니다.

**Llama 4**: Meta의 오픈 모델이지만, 가장 작은 Scout 모델도 109B 파라미터(총)로 SER5(16GB)에서 실행 불가능합니다. 라이선스도 700M MAU 이상 상업 사용 제한이 있어 Apache 2.0인 Gemma 4보다 불리합니다.

**Qwen 3.5**: 일부 벤치마크에서 Gemma 4보다 높지만, E4B처럼 5.5GB로 돌릴 수 있는 경량 모델이 없습니다. 26B MoE의 "3.8B만 활성화"라는 효율성도 Gemma 4만의 장점입니다.

### Gemma 4의 핵심 아키텍처 특징

**하이브리드 어텐션**: 슬라이딩 윈도우(로컬) + 글로벌 풀 어텐션을 5:1 비율로 교차 배치. 256K 토큰 컨텍스트를 지원하면서도 메모리 사용량을 억제합니다.

**Per-Layer Embeddings (PLE)**: E4B 모델의 비밀. 각 디코더 레이어마다 고유한 임베딩을 제공해서, 총 8B 파라미터이지만 4.5B만 활성화되면서 훨씬 큰 모델의 표현력을 확보합니다. 흥미롭게도 E4B가 큰 모델보다 할루시네이션 제어에서 더 높은 점수를 기록했습니다.

**MoE (Mixture of Experts)**: 26B A4B 모델은 128개 전문가 중 8+1개만 토큰당 활성화. 전체 26B 파라미터의 품질을 3.8B 활성 파라미터의 속도로 제공합니다.

**네이티브 function calling**: 모든 Gemma 4 모델에 도구 호출 기능이 내장. OpenClaw 에이전트 워크플로우와 자연스럽게 연동됩니다.

### Claude Max와의 역할 분담

Gemma 4는 Claude를 대체하는 것이 아닙니다. GPT-4o mini를 대체하는 것입니다.

| 역할 | 담당 | 이유 |
|------|------|------|
| 시스템 설계, 아키텍처 | Claude Max | Arena Elo ~1490, SWE-bench 79.6%, 대체 불가 |
| 복잡한 코드 작성 | Claude Max | LiveCodeBench 격차 (83% vs 52%) |
| 고품질 한국어 콘텐츠 | Claude Max | 브랜드 수준 콘텐츠는 타협 불가 |
| 반복적 자동화 (분류, 요약, 알림) | Gemma 4 | GPT-4o mini보다 우수 + 무료 |
| 데이터 수집 | Python 스크립트 | LLM 불필요, 직접 API 호출 |

## 3. SER5 전환 작업 요약

### 전환 전후 비교

```
[전환 전]                          [전환 후]
OpenAI GPT-4o mini (API)     →    Gemma 4 E4B (로컬, Ollama)
Brave Search API             →    Python 직접 API 호출 (무료)
LLM이 웹 검색해서 데이터 수집  →    Python이 수집 → Gemma 4가 가공
월 $30-40 유료 API 비용       →    ₩0

데이터 정확도: 75% 이하        →    99%+ (직접 API 호출)
```

### 전환 9단계

1. **백업 및 현재 상태 기록** — OpenClaw 설정, 크론잡 백업
2. **Ollama + Gemma 4 E4B 설치** — `ollama pull gemma4:e4b` (3.5GB)
3. **Gemma 4 동작 테스트** — 한국어 응답, API 서버, Docker 접근 확인
4. **바이블 데이터 수집 스크립트 작성** — CoinGecko, Yahoo Finance, FRED 직접 호출
5. **OpenClaw LLM 전환** — openclaw.json에서 GPT-4o mini → Gemma 4
6. **MSTR 브리핑 프롬프트 개편** — 웹 검색 방식 → JSON 데이터 기반 생성
7. **크론잡 재설정 + 통합 테스트** — 아침 7시 브리핑, 30분 추적
8. **유료 API 키 삭제** — 테스트 완료 후 OpenAI, Brave 키 삭제
9. **검증 및 문서 업데이트**

> 상세 작업 가이드: `SER5_Gemma4_전환_작업가이드.md` 참조

## 4. 크루즈AI 시스템에 미치는 영향

### 현재 크루즈AI 시스템 구성

크루즈AI는 5개 서브시스템으로 구성되어 있습니다:

- **CruiseStudio**: 블로그 콘텐츠 파이프라인 (Planner→Writer→Reviewer→Style→SEO)
- **CruiseKMS**: 지식 관리 시스템 (194개 네이버 블로그 → 구조화)
- **CruiseMD**: 선사 이메일 파싱 + 프로모션 관리
- **cruise-site**: Jekyll 기반 웹사이트 (cruise.co.kr)
- **CruiseCRM**: 고객 관리 (5,589건)

현재 이 시스템들은 Claude Max로 설계/개발 중이며, OpenClaw은 아직 적극 활용되지 않고 있습니다.

### Gemma 4 도입 후 변화 — 3계층 분업

SER5에서 Gemma 4가 안정적으로 돌아가면, 크루즈AI 시스템에도 동일한 3계층 분업이 적용됩니다:

### 크루즈 시스템 구체적 활용 시나리오

**시나리오 A: RSSC 상품 자동 클리핑 (매일)**

```
06:00  OpenClaw 브라우저가 rssc.com 크롤링       ← Python
06:05  수집 JSON을 Supabase에 UPSERT             ← Python
06:06  어제 대비 변동 감지 (단순 비교)             ← Python
06:07  변동 데이터를 Gemma 4에게 전달              ← Gemma 4
       → "이 변동이 유의미한가?" 판단
       → "SPL270410 Deluxe Suite $400 인하!" 알림 생성
06:08  텔레그램 발송                               ← Python
```

**시나리오 B: 이메일 프로모션 처리 (이벤트 트리거)**

```
IMAP 감지 → PDF 다운로드                           ← Python
→ Gemma 4: "이 이메일은 Type A/B/E 중 무엇?"      ← Gemma 4
→ Type A (Covert): covert_price 필드 UPSERT
→ Type B (Sail Into Savings): promo 필드 UPSERT
→ Type E (Removal): 상태를 removed로 변경
→ 변동 알림 텔레그램 발송                          ← Python
```

**시나리오 C: cruise.co.kr 상품 데이터 제공**

```
Supabase에 축적된 데이터 (크롤링 + 이메일 프로모션)
→ cruise.co.kr의 JavaScript가 Supabase REST API 조회
→ 가격 표시 우선순위: covert_price > promo_price > base_price
→ 프로모션 배지 자동 표시
→ 기항지/일정은 항상 크롤링 데이터 (가장 정확)
```

이 시나리오들의 공통점은 "Claude로 하기엔 아깝고, Python 스크립트만으로는 부족한" 작업을 Gemma 4가 채운다는 것입니다.

## 5. 아카이빙 시스템에 미치는 영향

### KMS 데이터 축적

현재 194개 네이버 블로그를 CruiseDB용으로 변환하는 작업이 예정되어 있습니다. 이 작업은 두 단계로 나뉩니다:

**구조 설계 (Claude Max)**: Markdown + YAML frontmatter 스키마, 태그 체계, 분류 기준 — 이건 한 번 잘 설계해야 후속 작업이 순조로운 영역입니다.

**일괄 변환 (Gemma 4)**: 설계된 스키마에 맞춰 194개 문서를 하나씩 변환하는 반복 작업 — 이건 Gemma 4가 로컬에서 무료로 처리할 수 있는 전형적인 계층 2 작업입니다.

### 향후 지식 축적 자동화

CruiseStudio가 블로그를 발행할 때마다 CruiseDB에 자동 저장되고, KMS에 지식이 축적되는 파이프라인에서도 Gemma 4가 "이 콘텐츠를 어떤 카테고리에 넣을 것인가", "관련 태그는 무엇인가" 같은 분류 작업을 담당할 수 있습니다.

## 7. 하드웨어별 사용 가능한 모델

### SER5 (현재 16GB)

| 모델 | 필요 RAM (4-bit) | 사용 가능 | 성능 수준 |
|------|:---------------:|:--------:|----------|
| Gemma 4 E2B | 4GB | ✅ | GPT-4o mini 동급 |
| Gemma 4 E4B | 5.5GB | ✅ 추천 | GPT-4o mini 상위 |
| Gemma 4 26B MoE | 16-18GB | ❌ | Claude Haiku급 |
| Gemma 4 31B | 17-20GB | ❌ | Sonnet 근접 |

### SER5 (32GB 업그레이드 후)

| 모델 | 필요 RAM (4-bit) | 사용 가능 | 성능 수준 |
|------|:---------------:|:--------:|----------|
| Gemma 4 E4B | 5.5GB | ✅ | 경량 보조 |
| Gemma 4 26B MoE | 16-18GB | ✅ 추천 | Claude Haiku급, 무료 |
| Gemma 4 31B | 17-20GB | ⚠️ 빠듯 | 최고 품질, 속도 느림 |

### SER8 (32GB, 미셋업)

SER8은 Ryzen 7 8845HS(내장 GPU 성능 우수) + 32GB로, Gemma 4 26B MoE를 더 빠르게 실행할 수 있습니다. SER5가 24시간 서버, SER8이 데스크톱/실험용이라는 역할 분담은 유지됩니다.

## 9. 핵심 인사이트

이번 전환에서 얻은 가장 중요한 교훈은 **"LLM에게 데이터를 가져오라고 시키지 말고, 데이터를 가져다 주고 판단만 시켜라"**입니다.

기존 방식(LLM이 웹 검색 → 데이터 수집 → 해석 → 브리핑 생성)은 모든 단계에서 오류가 발생할 수 있었습니다. 새 방식(Python이 API 직접 호출 → 검증된 데이터를 LLM에게 전달 → LLM은 판단/가공/문장 생성만)은 데이터 정확성을 보장하면서 LLM의 강점(언어 이해, 판단, 생성)에만 집중시킵니다.

이 원칙은 크루즈 시스템에도 그대로 적용됩니다:

- RSSC 가격 데이터 → OpenClaw 브라우저가 수집 → Gemma 4가 요약/알림 생성
- 이메일 프로모션 → Python IMAP이 수신 → Gemma 4가 타입 분류
- 고객 데이터 → Supabase에서 조회 → Gemma 4가 중복 감지/분류 보조

**데이터 수집은 Python이, 판단/생성은 Gemma 4가, 설계/복잡한 작업은 Claude Max가.** 이 3계층 분업이 비용은 최소화하고 자동화 범위는 최대화하는 구조입니다.
