# 📋 아카이빙 페이지 — 태그 체계 통합 설계서

> **작성일**: 2026년 4월 12일  
> **목적**: 기존 아카이빙 페이지(archive-v2)에 3차원 태그 체계를 통합하기 위한 설계 가이드  
> **배경**: 별도 채팅에서 "공부한 내용을 아카이빙하고, 태그로 나중에 실제 작업에 연결하고 싶다"는 요구에서 출발

---

## 1. 현재 상태 요약 (archive-v2)

### 잘 되어 있는 것 (유지)

| 항목 | 현재 구현 | 판단 |
|---|---|---|
| **탭 구조** | 수집 → 선별 → 발행완료 | ✅ 유지. 블로그 발행 워크플로우로서 잘 작동 |
| **소스 구분** | `claude-ai`, `claude-code`, `lilys`, `memo` + 컬러 바 | ✅ 유지. 시각적 구분이 효과적 |
| **2분할 레이아웃** | 좌: 카드 리스트 / 우: 디테일 패널 | ✅ 유지 |
| **검색** | 제목+태그+본문 전문 검색 + 하이라이트 | ✅ 유지 |
| **태그 편집** | 디테일 패널에서 추가/삭제 가능 | ✅ 유지하되 개선 필요 |
| **정렬** | 최신순/오래된순 토글 | ✅ 유지 |
| **다크모드** | 라이트/다크 전환 | ✅ 유지 |

### 부족한 것 (이번에 추가)

| 항목 | 현재 | 문제 | 개선 방향 |
|---|---|---|---|
| **태그 구조** | 단일 차원, 자유 입력 | 태그가 많아지면 의미 구분 불가 | 3차원 태그 체계 도입 |
| **"후속작업" 개념** | 없음 | 공부한 내용을 나중에 작업에 연결할 방법 없음 | 상태 태그로 관리 |
| **태그 필터 UI** | 전체 태그 한 줄 나열 | 태그 20개 넘으면 혼잡 | 차원별 그룹핑 |
| **프로젝트별 조회** | 없음 | "OpenClaw 관련만 보기" 불가 | 프로젝트 태그 필터 |

## 2. 태그 체계 설계 — 3차원

모든 아카이브 항목에 **각 차원에서 최소 1개씩** 태그를 붙인다.  
차원을 시각적으로 구분하기 위해 **프리픽스 규칙**을 사용한다.

### 차원 1: 프로젝트 태그 (어디에 속하는가)

**프리픽스: 없음 또는 `/`로 대분류/소분류 연결**

대분류와 소분류가 있되, 소분류는 필요할 때만 쓴다.

| 대분류 | 소분류 예시 | 설명 |
|---|---|---|
| `인프라` | `인프라/SER5`, `인프라/SER8`, `인프라/Docker` | 서버, 네트워크, 환경 설정 |
| `OpenClaw` | `OpenClaw/브리핑`, `OpenClaw/크론`, `OpenClaw/텔레그램`, `OpenClaw/브라우저` | OpenClaw 관련 전반 |
| `투자` | `투자/바이블`, `투자/MSTR`, `투자/InvestOS`, `투자/키움API` | 투자 모니터링, 자동매매 |
| `크루즈` | `크루즈/CruiseDB`, `크루즈/CruiseMD`, `크루즈/CruiseStudio`, `크루즈/CruiseKMS`, `크루즈/CruiseSNS`, `크루즈/CruiseCRM`, `크루즈/cruise-site`, `크루즈/VoyageClub`, `크루즈/RSSC`, `크루즈/Costa차터`, `크루즈/Adora`, `크루즈/LinkedIn` | 크루즈 비즈니스 |
| `로컬AI` | `로컬AI/Gemma`, `로컬AI/Ollama` | 로컬 모델 실험 |
| `아카이빙` | `아카이빙/파이프라인`, `아카이빙/블로그`, `아카이빙/SNS` | 이 아카이빙 시스템 자체 |
| `일반` | — | 특정 프로젝트에 속하지 않는 범용 학습 |

- 하나의 항목에 프로젝트 태그 **최대 3개**
- 소분류는 자유롭게 추가 가능 (사전 정의 필수 아님)

### 차원 2: 유형 태그 (무엇인가)

**프리픽스: `@`**

아카이브 항목의 성격을 나타낸다. **1개만 붙인다.**

| 태그 | 의미 | 예시 |
|---|---|---|
| `@정보분석` | 외부 콘텐츠(유튜브, 블로그, 기사) 분석 | 코드깎는노인 영상 분석 |
| `@작업로그` | Claude Code / 서버 작업 기록 | Gemma 4 전환 작업 |
| `@의사결정` | A vs B 비교, 아키텍처 선택 근거 | RSSC 크롤링 vs 이메일 파싱 |
| `@트러블슈팅` | 문제 발생 → 원인 → 해결 | Docker 네트워크 오류 수정 |
| `@설계` | 시스템/프로세스 설계 문서 | 바이블 브리핑 템플릿 설계 |
| `@발견` | 새로운 도구, 기술, 패턴 발견 | Gemma 4 MoE 아키텍처 특성 |
| `@회고` | 프로젝트 회고, 교훈 정리 | "자동화 ≠ 정확성" 교훈 |

### 차원 3: 상태 태그 (다음에 뭘 할 것인가)

**프리픽스: `!`**

이 항목이 액션을 필요로 하는지를 나타낸다. **1개만 붙인다.**  
이 차원이 "공부한 걸 나중에 작업에 연결한다"의 핵심이다.

| 태그 | 의미 | 전환 규칙 |
|---|---|---|
| `!후속작업` | 나중에 실제 작업에 반영할 것 | 작업 반영 시 → `!반영완료` |
| `!참고` | 당장 액션 없음, 배경지식으로 보관 | 필요 시 → `!후속작업`으로 승격 |
| `!반영완료` | 이미 작업에 반영함 | 최종 상태 |
| `!보류` | 검토했으나 현재 불필요 또는 시기상조 | 상황 변화 시 → `!후속작업` |

### 탭 vs 상태 태그 — 역할 구분

이 둘은 **별개의 축**이다. 혼동하지 않아야 한다.

| | 탭 (수집/선별/발행완료) | 상태 태그 (!후속작업 등) |
|---|---|---|
| **관리하는 것** | 블로그 발행 워크플로우 | 실제 작업 연결 |
| **질문** | "이걸 블로그에 쓸 건가?" | "이걸 나중에 작업에 반영할 건가?" |
| **예시** | 수집 → 선별 → 발행완료 | !후속작업 → !반영완료 |

하나의 항목이 탭은 `수집`(블로그에 아직 안 씀)이면서 상태 태그는 `!반영완료`(이미 작업에 반영함)일 수 있다. 이건 정상이다.

## 3. 태그 필터 UI 개선

### 현재 (단일 행)

```
[바이브코딩] [ClaudeCode] [RSSC] [Gemma4] [크롤링] [블로그자동화] ...
```

모든 태그가 한 줄에 섞여 있어서, 태그가 20개 넘으면 원하는 필터를 찾기 어렵다.

### 개선안 (차원별 그룹)

```
프로젝트  [OpenClaw] [투자] [크루즈] [로컬AI] [인프라] [아카이빙] [일반]
유형     [@정보분석] [@작업로그] [@의사결정] [@설계] [@발견] ...
상태     [!후속작업] [!참고] [!반영완료] [!보류]
```

- 각 차원이 별도 행으로 표시
- 차원 레이블(프로젝트/유형/상태)은 좌측에 작은 텍스트로
- 프로젝트 태그 클릭 시 소분류가 드롭다운 또는 인라인 확장으로 표시 (선택사항)
- **차원 간 필터는 AND**, **차원 내 필터는 OR**
  - 예: `OpenClaw`(프로젝트) + `!후속작업`(상태) = "OpenClaw 관련이면서 후속작업인 것"
  - 예: `OpenClaw` OR `투자`(프로젝트) = "OpenClaw 또는 투자 관련인 것"

## 4. 데이터 구조 변경

### 현재 archiveData 항목 구조

```javascript
{
  id: 1,
  status: "inbox",           // 탭 상태 (블로그 발행 워크플로우)
  source: "claude-ai",       // 소스 구분
  date: "2026-04-11",
  title: "...",
  excerpt: "...",
  summary: "...",
  body: "...",
  tags: ["블로그자동화", "아카이빙", "파이프라인설계"],  // 단일 차원
  links: { claude: "..." },
  memo: ""
}
```

### 개선안

```javascript
{
  id: 1,
  status: "inbox",           // 탭 상태 — 변경 없음
  source: "claude-ai",       // 소스 구분 — 변경 없음
  date: "2026-04-11",
  title: "...",
  excerpt: "...",
  summary: "...",
  body: "...",
  // ── 태그 3차원 분리 ──
  project: ["아카이빙/파이프라인"],           // 차원1: 프로젝트 (최대 3개)
  type: "정보분석",                          // 차원2: 유형 (1개)
  action: "후속작업",                        // 차원3: 상태 (1개)
  tags: ["LearningQueue", "블로그자동화"],   // 기존 자유 태그 (보조용, 유지)
  // ── links 확장 ──
  links: {
    claude_ai: "https://claude.ai/chat/xxx",      // Claude AI 대화 원본 링크
    claude_code: "세션ID 또는 export 파일 경로",    // Claude Code 세션 참조
    source: "https://youtube.com/...",              // 원본 소스 (유튜브, 기사 등)
    github: "https://github.com/...",               // 관련 코드
    blog: "https://...",                            // 발행된 블로그 URL
    related: ["https://claude.ai/chat/yyy"]         // 관련 다른 대화 링크
  },
  // ── 발행 메타데이터 (Phase 3 대비, 초기에는 빈 객체로 둬도 됨) ──
  publish: {
    blog_url: null,              // 발행된 블로그 URL
    blog_date: null,             // 발행일
    linkedin_draft: "",          // LinkedIn 초안 텍스트
    threads_draft: "",           // Threads 초안 텍스트
    linkedin_posted: false,      // LinkedIn 게시 여부
    threads_posted: false        // Threads 게시 여부
  },
  memo: ""
}
```

**변경 포인트:**
- `project`, `type`, `action` 필드 3개 추가
- 기존 `tags` 배열은 **보조 태그**로 유지 (프리픽스 없는 자유 키워드)
- 프리픽스(`@`, `!`)는 **UI 표시용**이며, 데이터에는 순수 텍스트로 저장
- `type`과 `action`은 문자열(단일 값), `project`는 배열(복수 가능)
- `links` 필드를 용도별로 세분화 (`claude_ai`, `claude_code`, `source`, `github`, `blog`, `related`)
- `publish` 객체 추가 — Phase 3(블로그→SNS 확장) 대비. 초기에는 빈 값으로 두되, 스키마에 미리 포함시켜 나중에 마이그레이션을 줄임

## 6. 핵심 사용 시나리오

### 시나리오 A: "OpenClaw 작업 재개 전, 관련 학습 내용 한번에 보기"

```
필터: project = OpenClaw + action = 후속작업
결과: OpenClaw 관련이면서 아직 작업에 반영 안 한 모든 항목
```

→ 작업 착수 전에 이 목록을 훑으면 맥락 손실 없이 시작 가능

### 시나리오 B: "이번 주 정보 분석한 것만 보기"

```
필터: type = 정보분석 + 날짜 정렬 최신순
```

→ 유튜브/블로그/기사 분석 결과만 모아보기

### 시나리오 C: "블로그에 쓸 만한 소재 찾기"

```
탭: 수집 또는 선별 + type = 작업로그 또는 의사결정
```

→ 작업 기록 중에서 스토리가 있는 것을 골라 블로그 글로 발전

### 시나리오 D: "투자 관련 모든 기록 보기"

```
필터: project = 투자 (소분류 무관)
```

→ 바이블, MSTR, InvestOS, 키움API 관련 전부

### 시나리오 E: "이미 반영한 것만 보기 (성과 확인)"

```
필터: action = 반영완료
```

→ 공부한 것 중 실제로 작업에 반영된 것. 학습 ROI 확인용.

## 7. 구현 우선순위

| 순서 | 작업 | 난이도 | 설명 |
|---|---|---|---|
| 1 | `archiveData` 구조에 `project`, `type`, `action` 필드 추가 | 낮음 | 데이터 스키마 변경 |
| 2 | 기존 10개 샘플 데이터에 새 필드 값 채우기 | 낮음 | 위 예시 참고 |
| 3 | 태그 필터 UI를 차원별 3행으로 변경 | 중간 | `renderTags()` 함수 수정 |
| 4 | `getFiltered()` 함수에 차원별 필터 로직 추가 | 중간 | AND/OR 로직 |
| 5 | 디테일 패널에 차원별 태그 표시 | 낮음 | `selectCard()` 함수 수정 |
| 6 | 태그 편집 UI에 차원 선택 추가 | 중간 | 드롭다운 or 프리픽스 자동 분류 |

## 8. 주의사항

- **기존 자유 태그(`tags` 배열)는 삭제하지 말 것.** 3차원에 딱 맞지 않는 키워드(도구명, 고유명사 등)를 담는 보조 역할로 유지한다.
- **프로젝트 소분류는 사전 정의를 강제하지 말 것.** 새 소분류는 자유롭게 추가하되, 대분류(인프라/OpenClaw/투자/크루즈/로컬AI/아카이빙/일반)는 고정한다.
- **태그 수 제한:** 프로젝트 최대 3개, 유형 1개, 상태 1개, 자유 태그 최대 5개. 총 10개를 넘지 않도록 한다.
- **프리픽스(`@`, `!`)는 UI 표시 전용.** 데이터에는 순수 텍스트로 저장하고, 렌더링 시 차원에 따라 프리픽스를 붙여서 표시한다.
