# 📥 아카이빙 시스템 — 입력 파이프라인 & 출력 확장 & 태그 고도화 설계

> **작성일**: 2026년 4월 12일  
> **작성 위치**: 크루즈인터내셔널 프로젝트 채팅  
> **목적**: 기존 archive-v2에 3가지를 추가하기 위한 설계서  
> 　　① **입력 자동화** — 작업 기록이 아카이빙에 자동으로 쌓이는 구조  
> 　　② **출력 확장** — 블로그 발행 후 LinkedIn/Threads 초안까지 연결  
> 　　③ **태그 고도화** — 단일 차원 태그 → 3차원 태그 체계  
>  
> **이 문서의 위치**: `아카이빙_페이지_설계서.md`(현재 기준 문서)를 **확장**하는 문서.  
> 기존 설계서의 구조(탭, 소스, UI, 데이터 구조)는 그대로 유지하고, 부족한 부분만 추가.

## 1. 전체 파이프라인 (기존 설계서 섹션 1 확장)

기존 설계서의 파이프라인에서 **입력단의 자동화 수준**과 **출력단의 SNS 연동**을 구체화한다.

```
[입력]                         [허브]                    [출력]

Claude Code 세션 ──(자동)──┐
                           │
Claude AI 대화 ──(반자동)──┼──→ 아카이빙 페이지 ──(선택)──→ 블로그 발행
                           │    (3차원 태그 필터)      (단일/복수)  ↓
Lilys AI 요약 ──(반자동)───┤    (검색 + 링크 보관)       LinkedIn 초안
                           │                            Threads 초안
수동 메모 ──(반자동)───────┘                              ↓
                                                    편집 → 게시
```

**기존 설계서와 달라진 점:**
- 입력 소스별 자동화 수준(자동/반자동/수동)을 명시
- Lilys AI: 수동 → **반자동** (Notion MCP 연동)
- 수동 메모: 수동 → **반자동** (작성 에이전트가 최적화 후 등록)
- 출력에 SNS 초안 생성 단계 추가
- 출력에 **복수 선택 발행** 지원
- 태그가 3차원으로 고도화

## 2. 입력 소스별 아카이빙 방법 (기존 설계서 섹션 6 확장)

기존 설계서 섹션 6에서 등록 트리거를 설계했지만, "실제로 어떤 기술로 어떻게 구현하는가"의 구체 조사가 부족했다. 이 섹션에서 보완한다.

### 2-1. Claude Code → 아카이빙 (자동)

**기존 설계서의 방식 (섹션 6.1 — CLAUDE.md 규칙):**
```
CLAUDE.md에 "작업 완료 후 archive.json에 항목 추가 + git push" 규칙 삽입
→ Claude Code가 작업 끝날 때마다 자동으로 아카이브 데이터 생성
```

이 방식은 **즉시 시작 가능**하고 추가 도구가 필요 없다. Phase 1에서는 이 방식을 쓴다.

**Phase 2에서 추가할 수 있는 고급 방식 (JSONL 파싱):**

Claude Code는 모든 대화를 `~/.claude/projects/` 디렉토리에 JSONL 파일로 자동 저장한다. 이 파일을 파싱하면 CLAUDE.md 규칙 없이도 자동 아카이빙이 가능하다.

검증된 도구들:
- **claude-code-log** (GitHub: daaain/claude-code-log) — JSONL → HTML 변환, TUI 세션 브라우징
- **claude-conversation-extractor** (GitHub: ZeroSumQuant) — JSONL → Markdown/HTML/JSON, 검색
- **cctrace** (GitHub: jimmc414/cctrace) — 세션 export + import, Git 연동

검증된 자동화 패턴 (DEV Community — jidong, 한국 개발자):
- 105개 세션 로그 → 자동 빌드 로그 블로그 포스트 생성 파이프라인
- 구조: `parse-sessions.py` → `generate-build-log.sh` (Claude CLI 연동) → git push → 자동 빌드
- cron 모드로 완전 자동화 가능

**두 방식의 관계:**

| | CLAUDE.md 규칙 (Phase 1) | JSONL 파싱 (Phase 2) |
|---|---|---|
| 구축 비용 | 0 (규칙 한 줄 추가) | 스크립트 개발 필요 |
| 트리거 | Claude Code가 작업 끝날 때 | cron 또는 hook |
| 스토리 품질 | Claude Code가 직접 작성 | 별도 Claude CLI가 가다듬음 |
| 누락 위험 | Claude Code가 규칙을 놓칠 수 있음 | JSONL에 100% 기록되므로 누락 없음 |
| **결론** | **지금 바로 시작** | **나중에 보완** |

### 2-2. Claude AI(claude.ai) 대화 → 아카이빙 (반자동)

**핵심 제약:** claude.ai에는 실시간 API가 없어서 프로그래밍적으로 대화를 가져올 수 없다.

5개 방법을 비교 조사한 결과:

| 방안 | 자동화 | 스토리 품질 | 원본 접근 | 일일 부담 | 판정 |
|------|:---:|:---:|:---:|:---:|:---:|
| 공식 내보내기 (Settings→Privacy→Export) | ❌ | ❌ 원본 그대로 | ✅ 전체 ZIP | 높음 | 월1회 백업용만 |
| Chrome 확장 (AI Chat Exporter) | ❌ | ❌ 원본 그대로 | ✅ PDF/MD | 중간 | 보조용 |
| **Claude에게 직접 요약 요청** | **△** | **✅ 가다듬어짐** | ❌ 요약만 | **낮음** | **✅ 메인** |
| **채팅 링크 함께 아카이빙** | **△** | ❌ 링크만 | **✅ 링크** | **낮음** | **✅ 위와 결합** |
| Claude Code Hook 연동 | ✅ | ✅ | ✅ | 거의 없음 | Phase 2 |

**최선책: "요약 요청 + 채팅 링크" 결합**

기존 설계서 섹션 6.2의 "아카이브에 추가해줘" 트리거와 동일한 개념이지만, **채팅 링크를 반드시 포함**하는 것이 핵심 차이점이다. 링크가 있으면 나중에 원본 대화로 바로 돌아갈 수 있어서, "claude.ai 검색이 약한 문제"를 보완한다.

```
대화 마무리 시 한 마디: "오늘 작업일지 정리해줘"
→ Claude가 요약 + 채팅 링크(https://claude.ai/chat/{uri}) 함께 생성
→ archive.json에 추가 (수동 복사 또는 Telegram 봇 경유)
```

**프로젝트 지침(Project Instructions)에 추가할 규칙:**
```
사용자가 "아카이빙해줘" 또는 "작업일지 정리해줘"라고 하면:
1. 이번 대화의 핵심 내용을 archive.json 형식으로 정리
2. 채팅 링크를 links.claude_ai에 포함
3. project/type/action 태그를 추천
4. 대화 중 첨부된 파일이 있었다면 links.attachments에 포함
5. 관련 다른 대화가 언급되었다면 links.related에 포함
```

### 2-3. Lilys AI → 아카이빙 (반자동 — Notion MCP 연동)

기존 설계서에서는 수동이었으나, **Notion MCP가 연결되어 있으므로 반자동**으로 업그레이드한다.

```
트리거: "Lilys 아카이빙해줘"
→ Notion MCP로 Lilys AI DB 조회 (ID: 33fab334-c1c0-8178-b191-e289cd2c7e66)
→ 최신 항목 또는 지정 항목의 내용을 가져옴
→ 작성 에이전트가 archive.json 형식으로 변환
→ 태그 자동 추천 포함
```

### 2-4. 수동 메모 → 아카이빙 (반자동 — 작성 에이전트 경유)

사용자가 대화 내용, 메모, 외부 정보를 자유롭게 입력하면 **작성 에이전트가 최적화한 뒤 등록**하는 구조.

**입력 방식:**
```
사용자: "이거 아카이빙해줘" + 내용 (텍스트, 링크, 첨부파일 등)
→ 작성 에이전트 처리:
  1. 내용을 구조화 (title, excerpt, summary, body 자동 생성)
  2. project/type/action 태그 자동 추천
  3. 첨부파일이 있으면 links.attachments에 경로/URL 포함
  4. 원본 링크가 있으면 links.source에 포함
  5. archive.json 형식으로 출력
→ 사용자 확인 후 등록
```

**작성 에이전트의 역할:**
- 날것의 입력을 받아서 **스토리 있는 글로 가다듬는 것**이 핵심
- 단순 복사가 아니라 "왜 이걸 했는지, 뭘 배웠는지, 다음에 뭘 할 건지"를 추론해서 정리
- 첨부파일(이미지, PDF, 문서 등)의 경로를 `links.attachments`에 배열로 저장

## 3. 태그 고도화 — 3차원 태그 체계 (기존 설계서 섹션 4 대체)

기존 설계서의 태그 시스템은 단일 차원 자유 입력이다. 태그가 20개 넘어가면 "이게 프로젝트 이름인지, 기술명인지, 상태인지" 구분이 안 되는 문제가 생긴다.

이를 **3차원**으로 분리한다. 기존 자유 태그(`tags` 배열)는 보조용으로 유지.

### 차원 1: 프로젝트 태그 — "어디에 속하는가"

하나의 항목에 **최대 3개**. 소분류는 `/`로 연결하며 자유롭게 추가 가능.

| 대분류 | 소분류 예시 |
|---|---|
| `인프라` | `인프라/SER5`, `인프라/Docker` |
| `OpenClaw` | `OpenClaw/브리핑`, `OpenClaw/크론`, `OpenClaw/텔레그램` |
| `투자` | `투자/바이블`, `투자/MSTR`, `투자/InvestOS` |
| `크루즈` | `크루즈/CruiseDB`, `크루즈/CruiseMD`, `크루즈/CruiseStudio`, `크루즈/CruiseKMS`, `크루즈/cruise-site`, `크루즈/CruiseCRM`, `크루즈/VoyageClub`, `크루즈/LinkedIn`, `크루즈/Costa차터`, `크루즈/Adora`, `크루즈/RSSC` |
| `로컬AI` | `로컬AI/Gemma`, `로컬AI/Ollama` |
| `아카이빙` | `아카이빙/파이프라인`, `아카이빙/블로그`, `아카이빙/SNS` |
| `일반` | — (특정 프로젝트에 속하지 않는 범용 학습) |

### 차원 2: 유형 태그 — "무엇인가" (1개만)

UI 표시 시 `@` 프리픽스. 데이터에는 순수 텍스트로 저장.

| 태그 | 의미 | 예시 |
|---|---|---|
| `정보분석` | 외부 콘텐츠(유튜브, 블로그, 기사) 분석 | 코드깎는노인 영상 분석 |
| `작업로그` | Claude Code / 서버 작업 기록 | Gemma 4 전환 작업 |
| `의사결정` | A vs B 비교, 아키텍처 선택 근거 | RSSC 크롤링 vs 이메일 파싱 |
| `트러블슈팅` | 문제 → 원인 → 해결 | Docker 네트워크 오류 수정 |
| `설계` | 시스템/프로세스 설계 문서 | 바이블 브리핑 템플릿 설계 |
| `발견` | 새로운 도구, 기술, 패턴 발견 | Gemma 4 MoE 아키텍처 특성 |
| `회고` | 프로젝트 회고, 교훈 정리 | "자동화 ≠ 정확성" 교훈 |

### 차원 3: 상태 태그 — "다음에 뭘 할 것인가" (1개만)

UI 표시 시 `!` 프리픽스. 데이터에는 순수 텍스트로 저장.
**"공부한 걸 나중에 작업에 연결한다"의 핵심.**

| 태그 | 의미 |
|---|---|
| `후속작업/즉시` | 바로 작업할 것 |
| `후속작업/중기` | 시스템 구축 중간 단계에 작업 |
| `후속작업/말기` | 시스템 완성 후 또는 마무리 단계에 작업 |
| `참고` | 당장 액션 없음, 배경지식으로 보관 |
| `반영완료` | 이미 작업에 반영함 |
| `보류` | 폐기 전 단계, 애매한 것 |
| `폐기` | 필요 없는 것 (최종 종결) |

**전환 규칙:**
```
후속작업/즉시 → 반영완료 (작업 완료 시)
후속작업/중기 → 후속작업/즉시 (시기 도래) → 반영완료
후속작업/말기 → 후속작업/중기 또는 /즉시 → 반영완료
참고 → 후속작업/* (필요 시 승격)
보류 → 후속작업/* (재검토 시) 또는 → 폐기
폐기 → (최종 상태. 되돌림 가능하되 기본적으로 종결)
```

**상태 변경 방식:**
- 디테일 패널에 **상태 드롭다운** 추가 — 사용자가 원하는 상태로 자유롭게 변경 가능
- 드롭다운 목록은 코드에 하드코딩하지 않고 **설정 가능한 배열**로 관리
- 후속작업 완료 시 "반영완료로 이동" 버튼을 별도로 표시 (빠른 전환용)
- 현재 archive-v2의 `moveStatus()` 함수는 탭 이동만 처리하므로, **action 변경용 `changeAction(id, newAction)` 함수를 새로 추가**해야 함

**필터 UI 반영:**
```
상태  [!즉시] [!중기] [!말기] [!참고] [!반영완료] [!보류] [!폐기]
```
- `!즉시`, `!중기`, `!말기`는 `후속작업`의 소분류로 표시
- 폐기 항목은 기본적으로 필터에서 숨김 처리 가능 (토글)

### 탭 vs 상태 태그 — 별개의 축

| | 탭 (수집/선별/발행완료) | 상태 태그 (!후속작업 등) |
|---|---|---|
| **관리하는 것** | 블로그 발행 워크플로우 | 실제 작업 연결 |
| **질문** | "이걸 블로그에 쓸 건가?" | "이걸 나중에 작업에 반영할 건가?" |

하나의 항목이 탭은 `수집`이면서 상태 태그는 `!반영완료`일 수 있다. 이건 정상이다.

### 태그 필터 UI 변경

**기존 (단일 행):**
```
[바이브코딩] [ClaudeCode] [RSSC] [Gemma4] [크롤링] [블로그자동화] ...
```

**변경 후 (차원별 3행):**
```
프로젝트  [OpenClaw] [투자] [크루즈] [로컬AI] [인프라] [아카이빙] [일반]
유형     [@정보분석] [@작업로그] [@의사결정] [@설계] [@발견] ...
상태     [!즉시] [!중기] [!말기] [!참고] [!반영완료] [!보류] [!폐기]
```

- **차원 간 필터는 AND**, **차원 내 필터는 OR**
- 예: `OpenClaw`(프로젝트) + `!후속작업`(상태) = "OpenClaw 관련이면서 후속작업인 것"

### 핵심 사용 시나리오

| 시나리오 | 필터 조합 |
|---|---|
| "OpenClaw 작업 재개 전, 관련 학습 내용 한번에 보기" | project=OpenClaw + action=후속작업 |
| "이번 주 정보 분석한 것만 보기" | type=정보분석 + 최신순 정렬 |
| "블로그에 쓸 만한 소재 찾기" | 탭=수집 + type=작업로그 or 의사결정 |
| "이미 반영한 것만 보기 (성과 확인)" | action=반영완료 |

## 5. 출력 확장 — 블로그 발행 → SNS 연동

### 현재
```
수집 → 선별 → 발행완료 (여기서 끝, 단일 항목만)
```

### 확장 후
```
수집 → 선별 → 발행완료
                    ↓
          단일 또는 복수 선택
                    ↓
               블로그 발행 (URL 생성)
                    ↓ (자동)
          LinkedIn 초안 + Threads 초안
                    ↓
            편집 페이지에서 수정
                    ↓
             게시 (복사→붙여넣기)
```

### 단일 발행 vs 복수 발행

| | 단일 발행 | 복수 발행 |
|---|---|---|
| **선택** | 카드 1개 클릭 → 발행 버튼 | 카드 여러 개 **체크박스** 선택 → "선택 발행" 버튼 |
| **작성 에이전트 처리** | 해당 항목만으로 블로그 글 생성 | 여러 항목을 **하나의 스토리로 엮어서** 생성 (예: "이번 주 작업 정리") |
| **SNS 초안** | 해당 글 기준 | 합쳐진 글 기준 |
| **archive.json 처리** | 해당 항목 status → published | 선택된 모든 항목 status → published, publish.blog_url 동일 URL |

### UI 변경 포인트

1. 카드 리스트에 **체크박스** 추가 (각 카드 왼쪽)
2. 체크박스 선택 시 상단에 **"N개 선택됨 | 선택 발행"** 액션 바 표시
3. 기존 단일 발행 버튼도 유지 (체크박스 없이 클릭→발행도 가능)
4. 복수 발행 시 작성 에이전트에게 선택된 항목들의 body/summary를 배열로 전달

### 작성 에이전트 복수 입력 처리

```
입력: [항목A의 요약+본문, 항목B의 요약+본문, 항목C의 요약+본문]
→ 에이전트 판단:
  - 공통 주제가 있으면 → 하나의 스토리로 엮음
  - 공통 주제가 없으면 → "이번 주 작업 모음" 형태로 구성
→ 출력: 블로그 글 1개 + LinkedIn 초안 + Threads 초안
```

### 발행 시 자동 생성되어야 하는 것

블로그에 글을 발행할 때, 해당 글의 핵심을 기반으로:
1. **LinkedIn 포스트 초안** — 영문, 크루즈 LinkedIn 포스팅 지침서의 톤과 형식, AEO 원칙 반영
2. **Threads 포스트 초안** — 한국어, 가볍고 친근한 대화체, 500자 이하

이 초안들은 디테일 패널 내 별도 영역 또는 탭에서 표시되며, 복사 버튼으로 바로 가져갈 수 있어야 한다.

→ 이 기능은 Phase 3에서 구현. Phase 1~2에서는 UI 프레임만 잡아두고 초안은 수동 생성.

## 6. 코드 변경 포인트 (archive-v2 기준)

### 변경이 필요한 함수들

| 함수 | 변경 내용 |
|---|---|
| `archiveData` 배열 | 각 항목에 `project`, `type`, `action` 필드 추가. `links`에 `attachments` 포함 새 키 추가. `publish` 필드 추가 |
| `renderTags()` | 기존: 전체 태그 한 줄 나열 → **변경: 3행 차원별 그룹** (프로젝트/유형/상태 각 행). 상태 행은 7개 태그 |
| `getFiltered()` | 기존: `activeTags`로 단순 OR → **변경: 차원별 필터 변수 분리, 차원 간 AND + 차원 내 OR** |
| `selectCard()` | 디테일 패널에 차원별 태그 표시 + **상태 변경 드롭다운** 추가. `publish` 필드 있으면 SNS 초안 영역 표시 |
| `renderList()` | 카드에 **체크박스** 추가 (복수 선택용) + 프로젝트 태그 1~2개 + 상태 아이콘 표시 |
| 태그 편집 UI | 기존 자유 태그 외에 차원 선택 드롭다운 추가 |
| **`changeAction(id, newAction)`** | **★ 신규:** 상태 태그 변경 함수. 드롭다운에서 선택 시 호출 |
| **`getSelectedIds()`** | **★ 신규:** 체크박스로 선택된 항목 ID 배열 반환 |
| **`publishSelected(ids)`** | **★ 신규:** 복수 항목을 한번에 published로 이동 |

### 변경하지 않는 것

- CSS 변수, 색상 체계, 폰트 — 전부 유지
- 탭 구조 (수집/선별/발행완료) — 유지
- 소스 분류와 컬러 바 — 유지
- 검색, 정렬, 다크모드 — 유지
- 메모 기능 — 유지
- 상태 이동 버튼 — 유지

## 8. 태그 운영 규칙

- 프로젝트 태그: 최대 3개
- 유형 태그: 1개 (필수)
- 상태 태그: 1개 (필수) — 7개 중 택1: `후속작업/즉시`, `후속작업/중기`, `후속작업/말기`, `참고`, `반영완료`, `보류`, `폐기`
- 기존 자유 태그: 최대 5개
- **총 10개를 넘지 않도록**

프리픽스(`@`, `!`)는 **UI 표시 전용**. 데이터에는 순수 텍스트로 저장하고, 렌더링 시 차원에 따라 프리픽스를 붙여서 표시.

대분류(인프라/OpenClaw/투자/크루즈/로컬AI/아카이빙/일반) 7개는 **고정**. 소분류는 자유롭게 추가 가능.

상태 태그 목록은 **설정 가능한 배열**로 관리. 사용자가 새로운 상태를 추가하거나 기존 상태를 수정할 수 있는 구조.
